# HAI Digital Twin

**Live dashboard:** [resonant-cobbler-711d31.netlify.app](https://resonant-cobbler-711d31.netlify.app)

A closed-loop digital twin of the HAI 23.05 industrial control system. The twin simulates normal and attack-injected plant behaviour across five control loops, and is used for attack detection and classification.

---

## Prerequisites

- [Conda](https://docs.conda.io/en/latest/miniconda.html) (Miniconda or Anaconda)
- Git

### 1 — Clone the repo

```bash
git clone <repo-url>
cd hai-digital-twin
```

### 2 — Create and activate the environment

```bash
conda env create -f environment.yml
conda activate digital_twin
```

> **Note:** The environment installs PyTorch via pip. If you need a CUDA-specific build, edit the `torch` line in `environment.yml` before creating the environment (see [pytorch.org](https://pytorch.org/get-started/locally/)).

### 3 — Verify raw data is in place

The following files must exist before running the pipeline:

```
00_data/processed/
├── train1.csv
├── train2.csv
├── train3.csv
├── train4.csv   ← held-out test period (contains attacks)
├── test1.csv    ← attack segments
└── test2.csv    ← attack segments
```

These files are the HAI 23.05 dataset. Download them from the [official HAI benchmark repository](https://github.com/icsdataset/hai) and place them in `00_data/processed/`.

## Running the Full Pipeline

All commands are run from the **repo root** (`hai-digital-twin/`) with the `digital_twin` conda environment active.

### Step 1 — Preprocess raw data

Converts the raw CSVs into normalised sliding-window `.npz` files under `outputs/scaled_split/`.

```bash
python 02_data_pipeline/scaled_split.py
```

> **Skip this step** if `outputs/scaled_split/` already contains `train_data.npz`, `val_data.npz`, and `test_data.npz`.

### Step 2 — Train the GRU digital twin

Training runs in two stages. Each stage warm-starts from the previous checkpoint:

```
outputs/pipeline/Re__reults_of_gru_after_wight_/gru_plant.pt  ← already in repo
        ↓
stage 1: train_gru_causal_plus.py       → outputs/pipeline/gru_causal_plus/
        ↓
stage 2: train_gru_scenario_weighted.py → outputs/pipeline/gru_scenario_weighted/
```

```bash
python 03_model/train_gru_causal_plus.py        # stage 1  (~hours on GPU)
python 03_model/train_gru_scenario_weighted.py  # stage 2  (~hours on GPU)
```

### Step 3 — Attack detection

Uses the `gru_scenario_weighted` checkpoint. Produces ROC curve, PR curve, confusion matrix, and detection stats.

```bash
python 05_detect/sec3_detection.py
# figures saved → report_plots/figures/s3/
```

### Step 4 — Attack classification

Runs three experiments (Baseline / TRTS / Mixed) and saves the final TRTS classifier.

```bash
python 05_detect/sec3_classification.py
# figures saved → report_plots/figures/s3c_*.png
# classifier  → outputs/classifiers/trts_rf_classifier.pkl
# scaler      → outputs/classifiers/trts_rf_scaler.pkl
```

## Architecture

```
[SP, PV history]  →  GRUController × 4 (PC, LC, FC, TC)  →  CV sequence
[TIT03, PP04SP]   →  CCSequenceModel                      →  pump speed
                                  │
                        GRUPlant (encoder–decoder)
                                  │
            Encoder: non-PV signals + scenario embedding → hidden h
            Decoder: autoregressive rollout
                     input_t = [ cv_target_t ‖ pv_{t-1} ]
                     pv_t    = FC( GRU(input_t, h) )
                                  │
                                  ▼
                  5 PV outputs: P1_PIT01, P1_LIT01, P1_FT03Z,
                                P1_TIT01, P1_TIT03
```

## Data

**HAI 23.05** — 1 Hz industrial control system logs with labelled attack segments.

| Split | Source files | Purpose |
|---|---|---|
| Train | train1 (100%), train2 (100%), train3 (first 30%) | Model training — normal only |
| Val   | train3 (last 70%) | Validation — normal only |
| Test  | train4 (100%) + 20% of attack segments | Evaluation |

Attack types: `AP_no` (Actuator Pollution, no combustion), `AP_with` (with combustion), `AE_no` (Actuator Emulation, no combustion).

Input window: 300 steps (5 min) → Target window: 180 steps (3 min), stride: 60 steps.

## Experiments & Development History

The `trials/` folder is a record of the full development journey — every model variant tried before arriving at the final architecture, in the order they were built.

---

**Chapter 1 — First attempts with sequence models**

We started by exploring whether standard sequence models could learn plant behaviour at all. The first serious attempt used an LSTM with causal structure baked into the input features — reasonable on normal data but struggled to generalise across attack scenarios. We then switched to a Transformer with scheduled sampling, which was slower to train and not meaningfully better than the LSTM, so it was dropped.

---

**Chapter 2 — Moving to GRU, building the causal backbone**

LSTM and Transformer weren't giving us the physical plausibility we needed. We rebuilt around a GRU encoder–decoder, starting with a plain baseline that confirmed a GRU could fit normal trajectories well. We then added causal graph guidance to the encoder inputs, which gave a clear improvement in physical consistency. Finally, we extended this with controller-in-the-loop inputs and a richer encoder — this version became the direct ancestor of the final model.

In parallel, we also built a standalone twin for a simpler boiler subsystem to stress-test the architecture on a different physical process before committing to the full HAI plant.

---

**Chapter 3 — Teaching the model about attacks**

Normal prediction was working. The challenge shifted to making the model scenario-aware. We first added a scenario embedding with separate output heads per attack type — promising but unstable without loss weighting. Adding class weights helped stability, but heads were still collapsing on minority attack types. A two-phase training curriculum (normal first, then attacks) improved stability but hurt generalisation. We then redesigned the attack heads with a cleaner separation between plant prediction and attack classification objectives, and refined the loss weighting — the best attack classification so far, but still not the final design.

---

**Final model**

The lessons from all chapters fed into a two-stage training scheme that combines the causal backbone from Chapter 2 with scenario-weighted loss from Chapter 3. The final model is a **regression model** — it predicts future PV trajectories (NRMSE = 0.0095 on held-out test data), and attack detection is derived from the residuals between predicted and observed behaviour (AUROC = 0.899).
